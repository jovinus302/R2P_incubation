## 1. 문제
"장소는 계속 저장하는데, 정작 그 상황이 오면 뭘 저장했는지 하나도 기억이 안 난다." 즐겨찾기·방문 이력이 수십~수백 개 쌓여도 "오늘 저녁 이 상황에 뭘 갈지" 결정 순간엔 무용지물 — 검증된 나만의 장소를 두고 다시 검색하거나 남의 추천에 의존.

## 2. 타깃 유저
지도 저장·방문 이력 수십~수백 개 쌓인 도시 생활자. 예: 서울 5년+ 거주, 즐겨찾기 100개+, 위치태그 외식·카페 사진 누적된 30대 직장인.

## 3. 제품 컨셉
상황 한 문장 ("금요일 저녁, 강남, 2시간, 부모님 동행") → 내 저장 장소·방문 이력·사진 속 장소를 엮어 동선·순서·이 상황에 맞는 이유를 카드로. 남의 추천 0개.

시나리오: ① 약속 잡히면 상황 한 줄 입력 ② 검증된 장소들로 짠 동선 카드 (순서·이동시간·상황 적합 이유) ③ 그대로 쓰거나 한 곳만 교체, 도착 순서 안내.

## 4. Why Now
저장 목록 + 방문 이력 + 사진 지오태그·캡션(비전)을 하나의 정합 서사로 엮는 추론이 frontier 능력으로 이제야 실용 수준. Ask Maps는 신규 추천형 대화 지도 — "내가 검증한 것"의 상황별 복원 아님.

## 5. 시장·경쟁 스냅샷
| 제품 | 회사 | 하는 일 | 갭 |
|---|---|---|---|
| Ask Maps | | 저장·이력 반영 대화 지도 | 신규 추천형 |
| Google Maps 저장 목록 | | 즐겨찾기 리스트 | 목록일 뿐, 실행 흐름 재구성 없음 |
| Wanderlog/Mindtrip/Plotline | 스타트업 | AI 여행 플래너 | 신규 추천·회고형, 일상 반복 상황 아님 |

so what: "검증된 개인 장소 + 실시간 상황 → 실행 동선" 직접 경쟁 미확인 (부재 증명 한계, 네이버/카카오 미조사).

## 6. 기술 실현성
가장 어려운 베팅: 저장 목록(위치·이름) + 방문 이력(시각·빈도) + 사진 EXIF·캡션을 오늘의 조건(시간·동행·목적)과 정합적으로 엮는 장문맥 추론. 확신 없음: "왜 좋았는지"까지의 사실적 복원 — 임포트 데이터에 그 이유가 없음 (7절).

**Frontier 필요성 입증 (레드팀 REJECT 대응)**: "시간·동행 조건 정렬·필터"만이면 규칙형으로 충분 — frontier가 필수인 지점은 다음 3개:
1. **암묵 신호 해석**: 방문 빈도·시간대 패턴·사진 속 장면(비전: 혼밥/모임/데이트 구분)에서 "이 장소가 어떤 상황용인지"를 추론 — 룰로 열거 불가한 비정형 판단
2. **동행 적합성 판단**: "부모님 동행"에 맞는 곳을 명시 태그 없이 장소 속성(소음·좌석·계단·메뉴)과 내 방문 맥락에서 추론
3. **동선 서사 조립**: 개별 장소가 아니라 "2시간 안에 말이 되는 순서" — 이동·체류·전환의 정합 계획
이 3개가 데모에서 룰 기반과 구분돼 보이지 않으면 "AI 한 스푼" 판정 수용 (kill 기준 연동).

## 7. 데이터 의존성 ★ (핵심 섹션 — 부활 서사)
- 첫 wow까지 필요 데이터: **Google Takeout (저장 장소 + Timeline 방문 이력) + 갤러리 EXIF 위치태그 — 일회성 임포트.**
- cold-start: **PARTIAL** — 시연자 실제 Takeout·갤러리 라이브 임포트로 첫 세션에 검증 장소 수백 개 확보.
- **round A kill과의 관계 (정직한 서사)**: 원안 U1은 "cold-start 실질 NO(위장 PARTIAL)"로 kill — 사유는 "'검증' 데이터가 임포트로 안 들어옴" (round 1 Wellness Coach 동형 함정). T1은 Takeout+EXIF라는 구체 wedge 지정으로 이 사유를 해소 — P1 명시 인정.
- **남은 한계**: 임포트가 주는 것은 "어디를 언제 갔는가"이지 "그때 왜 좋았는가"가 아님. 이유·감상은 데이터에 없음 → "이유를 복원한다"고 표현하면 confabulation.
- **제품 표현 조건 (P1 조건부 승인 핵심)**: "이유 복원"이 아니라 **"맥락 재구성"** — "왜 좋았는지 알려준다"가 아니라 "이 장소들이 오늘 상황(시간대·동행·소요시간)에 왜 맞는 조합인지 다시 짜준다"는 정직한 약속으로 한정.

## 8. 차별성
Ask Maps·저장 목록 대비: 저들은 신규 추천 또는 정적 리스트, 본 제품은 "이미 검증된 것"만 쓰며 상황별 실행 동선. moat 가설: **데이터** (개인 검증 이력의 정합 재구성) + **UX** (맥락 재구성이라는 정직한 프레이밍 자체가 신뢰의 원천).

## 9. 2주 MVP 스케치
- **데모 순간**: 시연자 실제 Takeout 라이브 임포트 → "금요일 저녁, 강남, 2시간, 부모님 동행" → 동선+맥락 카드 즉석 생성.
- **cut-list**: 사진 캡션(비전 설명문)은 사전 배치 처리 · 신규 도시·다중 동행 로직 스킵 · 실시간 교통은 고정값.
- **팀**: 엔지니어 2명 × 2주.
- **스크립트**: ① "제 Takeout 파일을 지금 올립니다" — 실데이터 신뢰 확보 ② 상황 입력 → 동선 카드 즉시 ③ "추천이 아니라 제가 가봤던 곳들을 오늘 상황에 맞게 다시 짠 것" — 맥락 재구성 표현으로 마무리.

## 10. 리스크
| 리스크 | 가능성 | MVP de-risk |
|---|---|---|
| Confabulation — "왜 좋았는지"를 지어낸 것처럼 보임 | **P1 high** | 부분 — "맥락 재구성" 표현 + 이유 텍스트 미제공으로 완화, 완전 해소 아님 |
| 공감이 hair-on-fire 아님 (편의성) — P2 | 중 | 안 함 — 반복 사용 절박함 검증 불가 |
| 프라이버시 — Takeout·위치 이력 민감 | 중 | 부분 — 일회성 로컬 임포트로 범위 제한 |
| Ask Maps의 저장 이력 재구성 확장 | 중 | 안 함 |

## 11. Go/No-Go 훅
- **성공 기준 (레드팀 반영 — 객관화)**: ① 실제 Takeout 라이브 임포트 2분 내 완료 ② **블라인드 평가**: 시연자 아닌 관찰자 5인에게 "본인 저장 이력 기반" 동선 카드 제시 → 과반이 "실제로 갈 법하다" 판정 + "지어낸 느낌" 지적 건수 0–1 ③ frontier 3지점(6절)이 룰 기반 대비 구분돼 보임
- **Kill 기준**: 1주차 임포트로 검증 장소 수백 개 확보 실패 시 중단. 블라인드 평가에서 "지어낸 느낌" 과반 지적 시 kill. 6절 frontier 3지점이 데모에서 룰 기반과 구분 안 되면 kill ("AI 한 스푼").
- **full go 가정**: "맥락 재구성" 프레이밍만으로 반복 상황(주말·손님·새 동네)마다 복귀 — 편의로도 리텐션 성립한다는 가정은 MVP 후 검증.
- **요청 결정 (레드팀 REJECT 반영 — 하향)**: ~~2주 MVP 승인~~ → **재리뷰 상신** — S3-T 경합(비ADVANCE) 지위 + 사용자(아이디어 제출자) 지명 3순위임을 명시. 위 객관화 기준 충족 가능성을 CTO 미팅에서 판단 요청. T6·T2와 동급 승인 요청 아님.
